# Develop — build, test, review, ship

The loop for building or changing a unit of code in any track: Build → Test → Review → address feedback, looping until review comes back clean, then Ship. Announce which step you're in; go wide within each (explore in parallel, run every check that fits, review across dimensions). The development flow (see `orchestrating-subagents`) uses this as its Develop phase — it is the `developer` agent's whole loop — but it applies just as much to a chore's code change, a backlog fix, or any standalone build — **scale the ceremony to the size of the change, never the gates**: "it's only N lines" never justifies skipping one; if a gate feels disproportionate, run it anyway and say why.

## Build

- **Entry check — never code ahead of the artifacts.** A new **or revised** contract — brand-new functionality, or a change to already-specced behavior or interfaces → `feature-spec` first; an approved spec not yet broken down → `feature-tasks`; a `ready` task, or a standalone fix that changes no contract → build here. (Which agent owns which phase: `orchestrating-subagents`.)
- **Plan the task at pick-up (task work) — the detailed plan is the Developer's, not the PM's.** First re-read the task's current AC from the **task doc** — the PM syncs it from Linear at hand-off, and the developer role carries no board tools. If the AC looks stale, contradictory, or thinner than the work implies, **say so in your report and stop short of guessing** — reconciling it is the orchestrator's to route (to the user, or to the PM for a fresh pull). Then fill the task doc's **Implementation** — ordered steps naming real files, grounded in the actual code first (read the files, map the full change/rename surface with grep before writing), changes shown as real code (diffs for reshapes, faithful sketches for new code, never pseudo-prose — `documentation-style`) — and its **Test plan** — each tier mapped to the AC it proves, folding new cases into existing expensive integration scenarios as sub-cases rather than minting new ones. Review the plan via `plannotator-loop` until a round returns clean, flip the task `planned`, then build. A task already `planned` in an earlier session: re-verify its AC and plan against the current code (and Linear) before building.
- **Write the code together with its tests — including a test of the *new behavior itself*, not just updates to the existing ones.** Existing tests passing only proves you didn't break the *old* behavior — never that the new behavior works. Every feature or behavior change ships a test that actually exercises the new path, at the **highest tier that can really run it** (e.g. a real-conductor sweettest that migrates v1→v3 end-to-end, not just the unit seam), dropping to a narrower or brand-new isolated test only when that tier genuinely can't reach the case. "I updated the existing tests and they're green" is necessary, never sufficient — name the test that proves the new behavior, or it isn't done. And cover **every tier the repo's conventions expect** for this kind of change (unit + integration + e2e where the repo has them): the existing test layout is the contract; the minimum viable test is not.
- Add a new cross-repo dependency only when the resource is genuinely shared.
- **Comment discipline — minimal, and only for the non-obvious.** A comment earns its place only when it explains *why* — a non-obvious intent, a trade-off, a gotcha, a cross-file contract — never *what* the code already says. Don't narrate (`// loop over sources`, `// return the package`), and don't write multi-line headers that re-argue the same point across functions: state each rationale **once**, where it's most relevant. If a block needs a paragraph to be understood, the fix is usually clearer code (better names, smaller functions), not the paragraph — heavy commenting is a smell that the code isn't self-explanatory. When in doubt, cut it.

## Test

- Run every check the environment allows — unit tests, typecheck, format, lint — and regression-check the existing tests. Respect repo conventions (e.g. run inside `nix develop -c` when the repo has a flake). Before declaring a heavy tier (e.g. a real-app UI suite) out of reach, actually try it — toolchains, displays, and drivers are usually present; only hand the user a command you genuinely cannot run.
- **Match the gates CI actually runs — open `.github/workflows/*` and run every job's checks, not a subset.** A passing `cargo test` is **not** a passing CI when CI also runs `cargo fmt --check` and `cargo clippy --all-targets -- -D warnings`; a passing `vitest` is not enough when CI also runs `typecheck` / `lint`. Format / lint / clippy failures are the classic silent miss — a test run never surfaces them, so "tests pass" is false confidence for "CI is green." If a job needs a toolchain you lack by default (e.g. a Holochain crate that won't build on a too-new nightly), install/use the one CI uses (the pinned stable, the repo flake) rather than skipping the gate and assuming it passes.
- **When your change edits a CI workflow file, lint the file itself before pushing** (`yaml-lint` / `actionlint`, or a YAML parser) — a malformed workflow doesn't fail a job, it schedules *no* jobs (GitHub shows a startup failure with no run), so green local tests say nothing about it. The classic trap: an unquoted `: ` (colon-space) in a step `name:` reads as a nested map and invalidates the whole file — quote any name/value that contains a colon.
- **Never read `$?` after piping a check into `tail` / `head` / `grep`** — the pipe's exit is the pager's (almost always 0) and masks the real failure. Capture the command's own status: `cmd; echo "EXIT=$?"` on its own line (or `${PIPESTATUS[0]}`), *then* truncate the output.
- **A gate that can't run as-is locally is a problem to solve, not a pass to assume.** Hunt the closest local proof before conceding: run the CI job's own script or container locally, use the repo's dry-run/sandbox mode, stub the credentialed dependency, or build a targeted harness that exercises the same path. Only when no local proof exists: state exactly what remains unproven and why, run everything that *can* run, and lean on the `/code-review` loop as the compensating check — never report a CI-only result as locally verified.
- **Fix pre-existing gate debt you surface — don't skip it.** When a gate (clippy `-D warnings`, fmt, a lint) is red on issues your change didn't cause — a stray unused import, dead code, a lint the base branch already violates — clear it in the same pass so the gate is actually green; a red gate you didn't cause still blocks the branch's CI, and "not my code" doesn't make the PR mergeable. Put an unrelated fix in its own clearly-scoped commit (`chore: clear pre-existing clippy debt`). Look before deleting code you didn't write: intentionally-kept items (documented stubs, deferred surfaces) get `#[allow(...)]` (or the language's equivalent) with the reason, not removal.
- Add the edge-case tests the implementation didn't handle, at the repo's established test tiers.
- **UI work follows the `ui-development` skill:** the journey spec declares visual beats, an env-gated `snap()` produces a numbered screenshot gallery of the real app, you self-review every shot, then the user reviews from the gallery and requests changes — regenerate per round until clean.
- **Everything CI runs that *can* run locally has passed locally before review — CI confirms, it never discovers.** The only exceptions are the no-local-proof gates named as unproven per the bullet above; any other red CI after push means this step was shortcut.

## Review

- **Actually invoke the `/code-review` slash command** (the Skill tool) on the diff, at **high effort or above** (`ultra` — the cloud multi-agent review — for a large or risky round) — do not substitute a hand-rolled multi-agent review of your own. The slash command is the gate; running your own finder agents "instead of" it does not count and is the wrong move even when it looks equivalent. (It is fine to *also* scout with subagents — when you fan more than one out, run them per `orchestrating-subagents`: self-contained briefs, no two writers in one tree, you consolidate — but the `/code-review` invocation must happen.) Triage its findings — fix the real ones, note false positives with why.
- **Security-sensitive diffs** — credentials/secrets, auth/authz, input parsing, crypto, network trust, permissions — also get **`/security-review`** (the actual slash command), its findings triaged the same way.
- **When the work has acceptance criteria (task work):** compare against them — is every AC met?
- **Stale sweep — always run it, and fix what it finds.** When the change replaced or renamed anything (routes, externs, types, error codes, config fields, spec files), grep the whole repo for every retired identifier and the old contract's distinctive phrases. Scope is everything, not just code: READMEs, AGENTS files, CI workflow comments, doc comments, deploy configs, dead spec links. The only legitimate hits are changelog *history* entries and tests that intentionally assert the old thing is gone; everything else gets fixed in the same change. (Leftovers that are another unit's scope — e.g. a UI that catches up later — must be named in the changelog/task, not silently left.)
- **Review is a loop of the whole cycle, not a one-time gate.** Every finding you fix is *new, unverified code*: a fix routinely introduces a fresh bug or only half-completes a sweep. So each round's fixes go **back through Build → Test (the gates rerun) → `/code-review`** on the post-fix diff, and the cycle repeats until a full round returns nothing actionable. **Non-negotiable for code you can't execute locally** (cloud/infra/deploy scripts, anything behind credentials or a live system): there the review *is* the verification gate tests can't be, so the fix→re-review loop is the only thing catching fix-induced regressions — budget for several rounds, not one. "Clean" = no new *actionable* findings; deferred-by-design items (reuse cleanups, out-of-scope) get filed to the backlog and **don't** reset the loop.
- **Pin each acted-on finding with a regression test where the logic is locally exercisable** (even via stubs), so the next regression trips `make test`, not the next reviewer.

## Ship — the Definition of Done

- Final **`/code-review` slash command** (invoked via the Skill tool — same rule as Review: the actual command, never a hand-rolled subagent review standing in for it) confirming the work is complete (every AC, for task work), with its findings triaged.
- **Exercise the changed flow end-to-end** (the `verify` skill) when the change has a runtime surface — drive the affected behavior in the real app, not only through tests, before the finalize commit.
- Changelog entry in each touched repo.
- **Published-crates check:** for every crate the change touched, determine whether it is published (full `[package]` metadata, prior release commits). For each one: append the changes to the crate's own `CHANGELOG.md` under `[Unreleased]`, verify packaging with `cargo publish --dry-run` (`--allow-dirty` before the commit), name the semver bump the changes imply (pre-1.0: breaking → minor), and tell the user it must be published before the future work that consumes it from the registry. The version bump itself stays a separate `chore(<crate>): release X.Y.Z` commit, per the repo's release precedent.
- **Backlog sweep** per the `backlog` skill: file the follow-ups this work surfaced, delete the items it resolved, and leave no P0 unscheduled.
- **When mirrored to Linear** (`linear-sync`): the issue moves to **Done** with the **as-built plan** pasted as a comment — a durable record, since the local task set is deleted once the whole feature ships. That's a Linear **write** — the `project-manager` agent's alone (only it carries the board-write tools): report ship-readiness so the orchestrator routes the Done-flip + as-built comment through it, never write the board from the developer role. The PM returns the planned diff, the orchestrator shows it to the user, and only then does it land (`linear-sync` § Discipline; the `guard-linear` hook prompts on the write either way) — shipping never mutates the shared board unseen.
- One `feat/fix/chore` branch per touched repo, and **one PR per branch** (the first approved push opens it, a later approved push updates it — never a second PR). **Publishing is gated (`core.md` § Publishing): show the user the commits + a diff summary + the exact target and get an explicit go-ahead for *that* push; never push, open/update a PR, or post anything on the user's behalf without it — and "ship it" / "address the review" / "make CI green" is never that approval.** (Local commits and branches need no approval — the push ask is where the work is reviewed.) The **PR body carries the delivered ticket/task and all the branch's related work** — AC, an as-built summary, the test(s) proving the new behaviour, deferred/backlog follow-ups, and cross-repo ship order — so the branch reviews as one self-contained PR. No AI/agent attribution.
- **Front-load the risks in the PR body — a short "review-focus / known-risks" note so review *confirms your flagged risks* rather than hunting blind.** Aim the reviewer straight at the load-bearing, uncertain, or hardest-to-verify parts of the change. Anything you *know* is provisional or unverified — a placeholder or locally-anchored value (a digest hashed from your own build rather than the published artifact), a CI-only path, a guard whose scope you're unsure is complete — is either **resolved before you open the PR** (verify against reality: fetch the real asset and hash *that*, run the real path, derive the guard's scope from the actual build command) or named as an **explicit blocker at the top of the body** — never buried in a file comment for the reviewer to discover. A reviewer burning effort to catch something you already knew was unverified is round-trips wasted on both sides; the symmetric fast path is *you pre-flag and resolve your known risks, they batch their pass into one round*. Applies to every reviewer — human or bot.
- **Address an external code review (CodeRabbit, a human reviewer) only through code** — fix the real findings in commits, note false positives in the commit/PR body, re-run the loop. **Never reply to a review thread, resolve it, or comment on the PR as the user** (`core.md` § Publishing); the fixes speak for themselves, and any written reply or re-review request is the user's to post. Push the review-fix commits under the same show-the-plan-then-approve gate.
- **When this is task work**, after merge: confirm the as-built behavior is captured in the spec docs (refresh `path:line` citations — no spec-vs-code drift), then flip the task to **`Status: completed ✅ (PR ref) — historical plan; as-built truth lives in the spec docs`** and tick its AC boxes. Completed tasks stay in `tasks/` as a frame of reference while sibling tasks are in flight — we complete tasks, we ship **features** — and the whole set is deleted only when the feature's tasks are all implemented.
- Task work moves **one task at a time** — after one ships, return to the catalog for the next (or back to spec/plan if the contract moved).
